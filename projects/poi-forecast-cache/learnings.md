# Learnings — poi-forecast-cache

Working ledger of patterns surfaced during this run. Reviewed at close-out; cross-cutting lessons migrate to durable surfaces, project-local ones drop with the folder.

1. **Shared dev DB across worktrees is a live foot-gun.** The dev `DATABASE_URL` had been advanced by the sibling `poi-auth` branch (BetterAuth tables); this branch's `db update` necessarily dropped them (PN has no additive-only mode — the planner reconciles live DB to contract). Surfaced to operator during D1. Candidate durable home: `drive/project/README.md` (one dev DB per worktree/branch, or coordinate).
2. **PN 0.14.0 `MIGRATION.MARKER_NOT_IN_HISTORY` + no `migrations/` dir → `db update` is the diagnostics-recommended flow.** Choosing `migration plan` would have baked sibling-branch table drops into committed history. Candidate durable home: `drive/plan/README.md` validation notes.
3. **Implementer persona references `.agents/rules/no-transient-project-ids-in-code.mdc`, which does not exist in this repo.** Both subagents fell back to the persona's inline ban list — worked fine, but the dangling reference should either be satisfied (add the rule file) or noted in `drive/` context. Candidate: note in `drive/plan/README.md § Known constraints`.
4. **Stable domain terms trip the transient-ID scan.** `ICON-D2` (official DWD model name) matches the `D[0-9]+` pattern; correctly exempted via the stable-reference carve-out. Pattern worth remembering for weather-domain repos.
5. **PN 0.14.0 gotcha candidates for `product-record-gotcha` routing at close-out** (no production impact this project): (a) ORM `.delete()` with an `.in([...])` predicate deletes one row per invocation, returning that row — multi-row predicate, single-row semantics; (b) `contract.d.ts` output types advertise branded `Timestamptz` strings, but the runtime lane returns `Date` objects.
6. **Spec bits can be lost in translation between altitudes.** The project spec's "extend staleAt instead of refetching" survived into the slice spec's design prose but not into the module surface it also specified (bare `Date` return) — the D3 wiring then faithfully implemented the deficient surface (reviewer finding F1). When a spec sentence names behaviour *and* a surface, check the surface can express the behaviour.
7. **PN 0.14.0 migration-CLI deviations surfaced by D4** (route to `product-record-gotcha` / upstream feedback at close-out, alongside item 5):
   - `migration plan --from ./path` and `ref set <name> ./path`: help text advertises `./path`, resolver rejects all paths (`PN-RUN-3000`; only hash/ref/migration-dir-in-graph accepted).
   - `migration new --from <hash>` silently ignores `--from` (scaffold records `from: null`, no warning).
   - `prisma-next-migrations` skill's free-import factory list is stale vs 0.14.0 (`createTable`/`createIndex`/`addForeignKey` are `Migration` instance methods; façade exports `col`/`fn`/`lit`/`primaryKey`/`foreignKey`/`unique`/`rawSql`/`createExtension`).
   - **Footgun:** `migration plan` on a dirty-ref state auto-writes a `baseline` package anchored at the refs index — anchored at the *destination* when the ref points there, which yields a destructive scratch plan (`dropTable` toward origin). `ref delete` first, then plan.
   - `migration plan` chains origin off the refs index, not the latest on-disk migration.
8. **Bootstrapping migration history onto a `db update`-initialized branch is a two-package dance**: `ref delete db` → emit origin contract → plan baseline (`null → origin`) → restore branch contract byte-exactly → plan edge (`origin → current`) → `ref set db <current>`. `MARKER_NOT_IN_HISTORY` then dissolves structurally (marker = edge's `to`) with no sign/resolve needed.
